## Abstract

R+2 (read: "R-plus-two") defines an open, cryptographically verifiable format for recording the actions of autonomous AI agents. Each action produces a signed receipt; receipts are linked by hash into per-agent chains; chains can be verified by any party without trusting the agent operator. The standard is designed to satisfy emerging regulatory requirements for AI audit trails (EU AI Act, India DPDP Act, UAE PDPL, US sectoral regulation) while remaining lightweight enough for high-volume agent deployments.

R+2 is a **format and protocol**, not a product. Anyone may implement it. No license fee applies. The reference implementation (`@trdnetwork/r2-verify`) is MIT-licensed.

## 1. Introduction

### 1.1 Motivation

AI agents — autonomous software entities that hold goals, manage state across interactions, and execute actions on behalf of human or institutional principals — are entering production deployments in finance, healthcare, government, and consumer software. As they take on responsibility for actions of material consequence, two questions become operationally urgent:

- **What did the agent do, when, and under whose authority?**
- **Can the answer be trusted by a party who does not trust the agent's operator?**

Existing audit-trail mechanisms (application logs, vendor dashboards, cloud-provider event streams) answer the first question but not the second. They produce records that the operator can modify after the fact, with no cryptographic guarantee of completeness or order. For regulated industries — and for any deployment where an external party (regulator, auditor, counterparty) may need to verify behaviour — that gap is dispositive.

R+2 closes the gap. Each agent action produces a signed, canonicalized receipt; receipts chain via hash so omission or reordering is detectable; chains anchor to an on-chain identity that cannot be silently rotated. Any third party can verify a chain without trusting the agent's operator.

### 1.2 Naming

The name **R+2** indicates the standard's position in a layered family:

- **R+1:** the underlying cryptographic primitives (Ed25519, SHA-256, RFC 8785).
- **R+2:** the receipt format, signing protocol, and chain construction defined in this document.
- **R+3:** optional regulator-grade audit export format (see [Section 11](#11-optional-layers)).
- **R+4 (future):** cross-jurisdictional reconciliation and reciprocity.

This document specifies R+2. It depends normatively on R+1 primitives and defines the optional R+3 layer non-normatively.

## 2. Conformance terminology

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**, **SHALL NOT**, **SHOULD**, **SHOULD NOT**, **RECOMMENDED**, **MAY**, and **OPTIONAL** in this document are to be interpreted as described in [RFC 2119] and [RFC 8174] when, and only when, they appear in all capitals as shown here.

An implementation that satisfies every **MUST** and **REQUIRED** clause of this document is **R+2-conformant**. An implementation that additionally satisfies optional clauses for sectoral profiles or R+3 export is **R+2-conformant + [profile]**.

## 3. Goals & non-goals

### 3.1 Goals

R+2 aims to:

1. Make the actions of an autonomous AI agent independently verifiable by parties who do not trust the operator.
2. Detect omission, reordering, or tampering of the agent's action history.
3. Bind an action to a specific agent identity, not merely to an operator.
4. Operate at scale (≥ 10,000 actions per agent per day) with low per-action overhead (< 1 ms signing cost, < 256 bytes receipt overhead).
5. Be implementable in any modern programming language using widely available cryptographic libraries.
6. Be free of patent claims and licensing fees.
7. Compose cleanly with regulatory frameworks (EU AI Act, DPDP, HIPAA, GDPR, MAS TRM, RBI guidance, UAE PDPL).
8. Support cross-vendor verification — a Claude agent and a GPT agent SHOULD both be verifiable by the same verifier using the same standard.

### 3.2 Non-goals

R+2 explicitly does **not** aim to:

1. Specify the agent's internal reasoning or chain-of-thought storage.
2. Specify any payment or transaction model. (See [DCS AI Treasury](https://dcslabs.ai/treasury) for an optional companion.)
3. Specify the substrate on which agent identity is rooted. Operators MAY use on-chain identity (e.g., a soul-bound token), Decentralized Identifiers (DIDs), corporate PKI, or other models. R+2 only requires that the identity be cryptographically resolvable to a public key.
4. Define a UI or dashboard for audit consumers. Implementations may provide such tooling; R+2 does not specify it.
5. Provide cryptographic privacy beyond the underlying primitives. R+2 receipts are designed to be shared with auditors who are authorized to see them; confidentiality is the operator's responsibility (e.g., via field-level encryption of `payload_hash` inputs before hashing).

## 4. Threat model

R+2 is designed to detect and resist the following adversaries:

| Adversary | Capability | R+2 response |
|---|---|---|
| **Tampering operator** | Can edit historical receipts after the fact | Hash-chain construction makes tampering detectable |
| **Selective-omission operator** | Drops inconvenient receipts from the chain | Chain integrity check shows missing predecessors |
| **Replay attacker** | Resubmits old receipts as new | Monotonic `action_index` + `timestamp` precludes replay |
| **Impersonating agent** | Mints a chain claiming to be another agent | On-chain identity binding requires holding the private key |
| **Compromised verifier** | A verifier returns false-positive results | Standard is implementable by anyone; cross-verification is RECOMMENDED for high-stakes audits |
| **Quantum-equipped adversary** | Future capability to break Ed25519 | Out of scope for v0.1.0; addressed in [Section 13.3](#133-post-quantum-roadmap) |

R+2 is **not** designed to resist:

- An adversary who controls both the agent's private key and the agent's runtime — they can produce arbitrary signed receipts indistinguishable from genuine ones. (This is the limit of any cryptographic audit system.)
- Denial-of-service against the verifier service.
- Out-of-band coercion of the operator to withhold receipts entirely.

For DoS resistance and operator-withholding scenarios, [R+3 audit export](#11-optional-layers) and federation are the recommended mitigations.

## 5. Architecture overview

```
   ┌──────────────────┐         ┌────────────────────┐
   │   AI Agent       │ signs   │   R+2 Receipt      │
   │  (action source) ├────────▶│   (canonical JSON  │
   │                  │         │    + signature)    │
   └────────┬─────────┘         └─────────┬──────────┘
            │                              │
            │  prev_hash                   │ chain append
            ▼                              ▼
   ┌──────────────────┐         ┌────────────────────┐
   │  Hash Chain      │◀────────┤  Receipt Store     │
   │  (per agent)     │         │  (operator-hosted) │
   └────────┬─────────┘         └─────────┬──────────┘
            │                              │
            │                              │  export
            ▼                              ▼
   ┌──────────────────┐         ┌────────────────────┐
   │  Verifier        │         │   R+3 Audit Bundle │
   │  (any party)     │         │  (optional, signed)│
   └──────────────────┘         └────────────────────┘
            ▲
            │
   ┌────────┴─────────┐
   │  Agent Identity  │
   │  (resolves to    │
   │   public key)    │
   └──────────────────┘
```

Per-agent state:

1. A **private signing key** (Ed25519, [RFC 8032]) held by the operator on the agent's behalf.
2. A **public verification key** published via the agent's identity (e.g., a DID document, an on-chain SBT, or a corporate certificate).
3. A **chain head** (the SHA-256 hash of the most recent receipt). Initial chain head is the SHA-256 of the agent identity URI.
4. A monotonically increasing **action_index** starting at 0.

## 6. Receipt schema

### 6.1 Required fields

Every R+2 receipt is a JSON object containing at minimum:

```json
{
  "version": "r+2/0.1.0",
  "agent_id": "did:dcs:base:0xbDd1...0f5F:42",
  "action_id": "uuid-v4-of-this-action",
  "action_index": 1234,
  "action_type": "memory.write | mcp.tool_call | treasury.transfer | ...",
  "timestamp": "2026-05-20T14:32:18.943Z",
  "principal": {
    "type": "human | system | agent",
    "id": "user:alice@example.com | system:scheduler-001 | did:dcs:..."
  },
  "context": {
    "session_id": "ulid-of-session",
    "parent_action_id": "uuid-or-null"
  },
  "payload_hash": "sha256:abc123...",
  "payload_schema": "https://schemas.dcslabs.ai/r2/payload/memory.write/v1",
  "chain_prev": "sha256:def456...",
  "chain_self": "sha256:789xyz...",
  "signature": "ed25519:base64(64 bytes)"
}
```

#### Field definitions

| Field | Type | Required | Description |
|---|---|---|---|
| `version` | string | yes | `"r+2/0.1.0"` for this version. Format: `r+N/major.minor.patch`. |
| `agent_id` | string | yes | Identity URI of the signing agent. Resolves to a public key. |
| `action_id` | string | yes | Globally unique identifier for this action (RECOMMENDED: UUID v4 or ULID). |
| `action_index` | integer | yes | Monotonic per-agent counter. First action = 0. |
| `action_type` | string | yes | Namespaced type. See [Section 16](#16-ianastyle-registry) for the registry. |
| `timestamp` | string | yes | RFC 3339 datetime, UTC, millisecond precision REQUIRED. |
| `principal` | object | yes | Who authorized the action. See 6.2. |
| `context` | object | yes | Session and parent-action linkage. See 6.3. |
| `payload_hash` | string | yes | SHA-256 of the canonicalized action payload (see 6.4). |
| `payload_schema` | string | yes | URI of the payload's JSON Schema. |
| `chain_prev` | string \| null | yes | SHA-256 of the previous receipt's canonical form. `null` only when `action_index == 0`. |
| `chain_self` | string | yes | SHA-256 of this receipt's canonical form *excluding* the `signature` field. |
| `signature` | string | yes | Ed25519 signature over `chain_self`'s underlying bytes. Format: `"ed25519:<base64>"`. |

### 6.2 Principal sub-schema

```json
{
  "type": "human | system | agent",
  "id": "<scheme-specific identifier>",
  "authority": {
    "method": "oauth | api_key | mtls | sbt | other",
    "issued_at": "RFC 3339 datetime",
    "expires_at": "RFC 3339 datetime | null",
    "delegation_chain": ["did:dcs:...", "..."]
  }
}
```

`authority` is REQUIRED when the principal is `human` or `agent`. For `system` principals, it MAY be omitted.

### 6.3 Context sub-schema

```json
{
  "session_id": "string identifier for the agent session",
  "parent_action_id": "uuid of immediate parent action, or null",
  "originating_action_id": "uuid of the root action that originated this chain, or null",
  "regulatory_scope": ["dpdp", "gdpr", "hipaa", "..."]
}
```

`regulatory_scope` MAY be empty. Sectoral profiles ([Section 12](#12-sectoral-profiles)) define what regulatory frameworks SHOULD be tagged for which action types.

### 6.4 Payload

The action's actual payload (the memory write, the tool-call arguments, the transaction details) is NOT included in the receipt. Only its SHA-256 hash is, with the schema URI for verification. Rationale:

- Receipts MUST be short enough to embed in audit logs at scale.
- Payloads MAY contain personal data; hashes don't.
- The payload is verifiable independently by anyone who holds the payload itself plus the schema.

A separate store (the "payload sidecar") holds payloads, indexed by `action_id`. The store can be operator-hosted, customer-hosted, or escrowed. R+2 does not specify the store; it only requires that the hash linkage be sound.

## 7. Signing & canonicalization

### 7.1 Canonicalization

All hashing and signing MUST use **JSON Canonicalization Scheme (JCS)** as defined in [RFC 8785]. JCS produces a deterministic byte representation given the same JSON object regardless of property ordering, whitespace, or encoding implementation differences.

Implementations MUST NOT use ad-hoc canonicalization (sorting keys themselves, custom whitespace removal, etc.). They MUST use a vetted JCS implementation. Failure to canonicalize correctly produces signatures that are valid by chance but unverifiable across implementations.

### 7.2 Hash function

R+2 v0.1.0 uses **SHA-256** ([FIPS 180-4]) for all hashes, encoded as lowercase hexadecimal prefixed with `sha256:`.

Future versions may add additional hash functions (e.g., SHA-3, BLAKE3). Versioning is handled per [Section 13](#13-versioning-policy).

### 7.3 Signature algorithm

R+2 v0.1.0 uses **Ed25519** ([RFC 8032]) for all signatures, encoded as Base64 (standard, not URL-safe) prefixed with `ed25519:`.

Ed25519 is REQUIRED because:

1. It is deterministic — no random nonce required, removing a common source of vulnerability.
2. Signatures are short (64 bytes raw, 88 bytes base64) — important at agent-action volume.
3. Verification is fast and constant-time.
4. Library support is universal in modern languages.

Implementations MUST NOT substitute ECDSA, RSA, or other algorithms in an R+2-conformant receipt.

### 7.4 Signing procedure

To produce a signed receipt:

1. Construct the receipt JSON with all REQUIRED fields except `chain_self` and `signature`.
2. Compute `chain_self`: SHA-256 of the canonical JSON of the receipt with `chain_self` and `signature` set to `null`. Set `chain_self` to the resulting hash.
3. Compute `signature`: Ed25519 sign the **raw bytes** of the canonical JSON of the receipt with `signature` set to `null` (but `chain_self` set to its computed value). Encode as `ed25519:<base64>`.
4. Emit the receipt.

### 7.5 Verification procedure

To verify a signed receipt:

1. Parse the receipt.
2. Compute the expected `chain_self`: canonical JSON with `chain_self` and `signature` set to `null`, then SHA-256. MUST equal the receipt's `chain_self`.
3. Resolve `agent_id` to a public key via the identity layer ([Section 9](#9-identity-model)).
4. Verify `signature` over the canonical JSON with `signature` set to `null`. MUST validate.
5. If `chain_prev` is non-null, the previous receipt's `chain_self` MUST equal this receipt's `chain_prev`.
6. `action_index` MUST equal the previous receipt's `action_index + 1` (or `0` for the first receipt).

Any step's failure renders the chain non-verified at that point. Implementations MUST return precise failure reasons; "verification failed" with no detail violates the user's right to audit clarity.

## 8. Hash chain construction

### 8.1 Per-agent chain

Every R+2 agent maintains exactly one chain. The chain is identified by the agent's identity URI. Branching is NOT permitted in v0.1.0.

### 8.2 Genesis

For `action_index == 0`:

- `chain_prev` MUST be set to `sha256:<hex>` where the hex is the SHA-256 of the UTF-8 bytes of the `agent_id` string. This binds the chain origin to the identity.

### 8.3 Append rules

For each subsequent receipt:

- `chain_prev` MUST equal the SHA-256 of the canonical form of the immediately preceding receipt (with `signature` set to `null`).
- `action_index` MUST equal preceding `action_index + 1`.
- `timestamp` MUST be greater than or equal to the preceding receipt's `timestamp`. Strictly greater is RECOMMENDED.

### 8.4 Detection of tampering

- A modified receipt produces a different SHA-256, breaking the next receipt's `chain_prev`.
- An omitted receipt creates a gap in `action_index` and breaks `chain_prev` of the next surviving receipt.
- A re-ordered receipt fails both `chain_prev` and `action_index` checks.

### 8.5 Recovery from chain breakage

In v0.1.0, broken chains cannot be repaired. The break is permanent and visible to all verifiers. Operators experiencing a chain break MUST document the cause as part of their compliance posture and SHOULD start a new chain (with a new `agent_id` if regulatory scope requires).

Future versions MAY define a "chain reset" receipt with cryptographic proof of operator authority. Out of scope for v0.1.0.

## 9. Identity model

R+2 requires that every `agent_id` resolve to an Ed25519 public verification key. Three resolution mechanisms are SUPPORTED:

### 9.1 On-chain SBT (recommended)

For deployments using DCS-compatible infrastructure:

- `agent_id` format: `did:dcs:<chain>:<contract>:<token_id>` (e.g., `did:dcs:base:0xbDd1f5fC349D9a8EfCEb07Edbd491233b2540f5F:42`).
- Resolution: read the token's `pubkey` field from the contract on the named chain.
- Recommended chain: Base mainnet via the TRDWorkerSBT contract.

### 9.2 W3C DID

For DID-using deployments:

- `agent_id` format: any valid DID URI (e.g., `did:web:agents.example.org:42`).
- Resolution: per the DID method specification.
- Verification key MUST be of type `Ed25519VerificationKey2020` or equivalent.

### 9.3 Corporate PKI

For closed deployments:

- `agent_id` format: `pki:<issuer>:<serial>` (e.g., `pki:enterprise-ca-2026:00045a`).
- Resolution: out-of-band via X.509 certificate lookup.
- The certificate MUST encode an Ed25519 public key.

Operators MAY use any of the three, but MUST publish the resolution mechanism in their conformance statement.

## 10. Verifier requirements

An **R+2 verifier** is software that, given a receipt and access to the chain history and identity resolution, produces a definitive verdict on whether the chain is valid.

### 10.1 MUST behaviours

A verifier MUST:

1. Implement RFC 8785 JCS canonicalization correctly (test vectors in `/spec/test-vectors/`).
2. Verify Ed25519 signatures using a constant-time implementation.
3. Compute SHA-256 hashes per FIPS 180-4.
4. Walk the chain backward from any receipt to verify all `chain_prev` links.
5. Verify `action_index` monotonicity.
6. Verify `timestamp` monotonicity (non-decreasing).
7. Resolve `agent_id` via the operator's stated resolution method.
8. Return a structured verdict: `{valid: bool, failed_at: action_id|null, reason: string|null}`.

### 10.2 SHOULD behaviours

A verifier SHOULD:

1. Cache resolved public keys to avoid repeated identity lookups.
2. Support parallel verification of independent chains.
3. Provide a stream/batch mode for verifying large chain segments.
4. Expose Prometheus-compatible metrics: receipts_verified_total, verification_duration_seconds, verification_failures_total.

### 10.3 Reference implementation

The reference verifier is `@trdnetwork/r2-verify` (npm), maintained by DCS Labs. It is MIT-licensed. Implementers SHOULD cross-check against it. Test vectors in this repository (`/spec/test-vectors/`) provide additional conformance evidence.

## 11. Optional layers

### 11.1 R+3: regulator-grade audit export

R+3 defines a packaged export format suitable for submission to a regulator or external auditor:

- Contains a contiguous range of receipts plus their payload sidecars.
- Wrapped in a top-level signed envelope by the operator.
- Includes a manifest listing all included receipts, schemas, and identity proofs.
- Includes a chain-of-custody record.

R+3 is OPTIONAL but RECOMMENDED for any deployment subject to regulatory audit obligations. Specification: separate document at `/spec/r+3-audit-export.md`.

### 11.2 Federation

R+2 chains MAY federate across operators. Federation allows a multi-operator agent (e.g., a customer support agent that spans two SaaS vendors' systems) to maintain a single coherent chain. Federation is OPTIONAL. Specification: separate document at `/spec/federation-v1.md`.

## 13. Versioning policy

### 13.1 Semantic versioning

R+2 follows semantic versioning: `r+2/MAJOR.MINOR.PATCH`.

- **MAJOR**: breaking changes to receipt schema, signing protocol, or chain construction. Increment requires a 6-month transition window.
- **MINOR**: backward-compatible additions (new OPTIONAL fields, new action types, new sectoral profiles).
- **PATCH**: editorial corrections, clarifications without semantic change.

### 13.2 Backwards compatibility

A v0.1.x verifier MUST accept v0.1.y receipts for any y ≥ x (forward compatibility within minor). A v0.1.x verifier MAY reject v0.2.x receipts.

Operators upgrading to a new MINOR version MUST NOT mix versions within a single chain; if they upgrade, they SHOULD start a new chain or migrate the whole chain.

### 13.3 Post-quantum roadmap

R+2 v0.1.0 uses Ed25519, which is not quantum-resistant. The current consensus is that practical quantum attacks on Ed25519 are 10–15 years away.

R+3.0 (target: 2028) will introduce a hybrid signing mode: Ed25519 + ML-DSA (NIST PQC). Operators wishing to begin migration early MAY emit dual signatures non-normatively today. Full PQ transition is on the standard roadmap.

## 14. Security considerations

### 14.1 Key compromise

If an agent's private signing key is compromised, all subsequent receipts signed by that key are suspect. Operators MUST:

1. Revoke the compromised identity at the identity layer (e.g., burn the SBT, revoke the DID, revoke the cert).
2. Issue a notification to verifiers via the operator's status channel.
3. Begin a new chain with a new identity.

R+2 v0.1.0 does NOT define a revocation list. This is intentional — revocation lists create centralization. The identity layer SHOULD provide revocation primitives.

### 14.2 Side-channel attacks

Constant-time Ed25519 implementations are REQUIRED. Implementations using non-constant-time signing are vulnerable to timing attacks that leak the private key.

### 14.3 Replay across chains

A receipt is bound to a specific chain by `chain_prev`, `action_index`, and the signature. It cannot be replayed onto a different chain without forging the signature.

### 14.4 Operator collusion

R+2 does not protect against an operator who simply never emits receipts for some actions. Mitigation: the principal SHOULD require receipt-of-receipts as a counter-signed acknowledgement before payment or settlement.

## 15. Privacy considerations

### 15.1 Payload hashing

R+2 receipts contain only the SHA-256 hash of the payload, not the payload itself. This means a verifier can confirm an action occurred without seeing personal data.

### 15.2 Selective disclosure

Operators MAY redact payloads in disclosure to particular auditors. As long as the payload hash is computable from the original, an auditor can verify any specific payload they receive.

### 15.3 Right to erasure

Under GDPR Article 17 / DPDP Section 12, data subjects have a right to erasure. R+2 supports this without breaking the chain by:

1. Erasing the payload sidecar entry for the affected action.
2. Recording a separate `action_type: "data.erasure"` receipt referencing the original action.

The hash in the original receipt remains; the underlying payload no longer exists. This is GDPR/DPDP-compliant erasure.

### 15.4 Sensitive `payload_hash` inputs

If the payload contains sensitive data (medical records, financial details), operators MAY field-encrypt before hashing. The hash is then over the ciphertext, not the plaintext. A verifier who decrypts the payload separately can still confirm the hash matches.

## 18. Glossary

**Agent**: An autonomous software entity that holds goals, manages state across interactions, and executes actions on behalf of a principal.

**Canonicalization**: A deterministic byte-level representation of a JSON object. R+2 uses RFC 8785 JCS.

**Chain**: The ordered sequence of receipts for a single agent, linked by hash.

**Chain head**: The hash of the most recent receipt in a chain.

**Federation**: A protocol (R+2 optional layer) for multi-operator chains.

**Operator**: The entity that hosts the agent runtime and holds the agent's private key.

**Principal**: The entity that authorizes the agent's actions (a human, another agent, or a system process).

**Receipt**: A signed, canonicalized JSON object representing a single agent action.

**Sectoral profile**: A constraint document specifying R+2 usage for a particular regulatory framework.

**SBT**: Soul-bound token — a non-transferable on-chain identity (ERC-5114).

**Verifier**: Software that determines whether a chain is cryptographically valid.
